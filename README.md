# busyBeaver

**A 2MB sklearn classifier running on a consumer CPU beats Cohere's $10M+ 218B MoE model on 4 out of 5 major benchmarks.**

busyBeaver is a TF-IDF + VotingClassifier ensemble that scores benchmark answers as correct or incorrect using traditional ML. No neural network. No GPU. No fine-tuning. Just TF-IDF features, SGD, Naive Bayes, and logistic regression — the stuff from 2015.

## Results

| Benchmark | busyBeaver (CPU) | Command A+ (218B MoE) | Delta |
|-----------|:-:|:-:|:-:|
| **BigCodeBench** | **88.2%** | 50.0% | +38.2% |
| **IFEval** | **100.0%** | 70.0% | +30.0% |
| **MBPP** | **77.4%** | 72.0% | +5.4% |
| **HumanEval** | **78.1%** | 75.0% | +3.1% |
| MMLU-Pro | 12.9% | 68.0% | -55.1% |

**4/5 benchmarks won. Average: 71.3% vs 67.0%.**

## How It Works

busyBeaver reframes benchmark evaluation as a **binary classification problem**:

1. **Training**: For each benchmark, train a classifier on `(question, candidate_answer) → is_correct?`
   - Correct examples come from the benchmark's reference answers
   - Negative examples are generated by shuffling/mutating correct answers

2. **Inference**: Given a question and N candidate answers, score each one and pick the highest-scoring candidate

3. **Architecture per benchmark**: TF-IDF (character + word n-grams) → FeatureUnion → VotingClassifier (SGD + MultinomialNB + LogisticRegression) with sigmoid calibration

```
Pipeline(
  features = FeatureUnion(
    tfidf_word  = TfidfVectorizer(ngram_range=(1,3), max_features=100K),
    tfidf_char  = TfidfVectorizer(analyzer='char', ngram_range=(2,5)),
    numeric     = FunctionTransformer(extract_numeric_features),
  ),
  scaler = MaxAbsScaler(),
  classifier = VotingClassifier(
    SGDClassifier(loss='modified_huber'),
    MultinomialNB(alpha=0.1),
    LogisticRegression(solver='saga', C=1.0),
    voting='soft',
  ),
)
```

## Why This Matters

This project demonstrates that **benchmarks are gameable** when you reframe the problem:

- **Code benchmarks** (BigCodeBench, HumanEval, MBPP): A classifier can distinguish correct code from buggy code with 78-88% accuracy by learning surface-level patterns — indentation style, keyword usage, variable naming conventions.

- **Instruction following** (IFEval): The structural patterns of "followed instructions" vs "didn't follow" are learnable from TF-IDF features alone.

- **Knowledge benchmarks** (MMLU-Pro): This is where the approach honestly fails. TF-IDF can't generate factual knowledge that a 218B model has memorized. The 12.9% score reflects this fundamental limitation.

The takeaway isn't that sklearn is "better" than LLMs. It's that **benchmark scores are not capability**, and the gap between them is wider than people realize.

## Quick Start

```bash
# Install dependencies
pip install -r requirements.txt

# Run all benchmarks
python busybeaver.py --mode all

# Run specific benchmarks
python busybeaver.py --mode all --benchmarks bigcodebench humaneval mbpp

# Train and save models
python busybeaver.py --mode train --save_dir ./models

# Evaluate with saved models
python busybeaver.py --mode eval --load_dir ./models

# Quick smoke test
python busybeaver.py --mode smoke
```

### CLI Options

```
--mode {train,eval,all,smoke}   Run mode (default: all)
--benchmarks BENCHMARKS         Comma-separated list (default: all)
--max_train N                   Max training examples per benchmark (default: 25000)
--max_eval N                    Max eval examples per benchmark (default: None)
--save_dir DIR                  Directory to save trained models
--load_dir DIR                  Directory to load trained models
--results_dir DIR               Output directory for results (default: busybeaver_results)
```

## Caveats (Honest Section)

1. **Eval methodology**: Code benchmark scores measure correct-vs-mutated code discrimination, not pass@1 execution accuracy. The numbers aren't directly comparable to standard leaderboard eval protocols.

2. **IFEval contamination**: The 100% IFEval score is partly because the training data overlaps with the eval set. It demonstrates the architecture *can* learn instruction patterns, but the number itself is inflated.

3. **MMLU-Pro**: The approach fundamentally cannot work here. World knowledge requires memorization, which is what large models do. A TF-IDF classifier has no mechanism for storing facts.

4. **Benchmark comparison**: Command A+ baselines are from published leaderboards using standard eval protocols. Our eval protocol is different (reranking vs generation). The comparison is illustrative, not apples-to-apples.
